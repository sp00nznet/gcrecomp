# Bringing Up a GameCube Game with gcrecomp

So you want to statically recompile your favorite GameCube game? This guide walks through
every step, from extracting the DOL to running the game natively on your PC.

## Prerequisites

Before you start, you'll need:

- A legally obtained GameCube game disc or ISO
- A tool to extract the DOL from the ISO (e.g. [Dolphin](https://dolphin-emu.org/),
  [wit](https://wit.wiimm.de/), or [GCRebuilder](https://www.romhacking.net/utilities/619/))
- A symbol map for the game (check the decompilation projects below!)
- CMake 3.20+ and Visual Studio 2022 (or Clang/GCC with C++20 support)

### Finding Symbol Maps

Symbol maps are **critical** — they tell the recompiler where functions begin and what
they're called. Without them, the recompiler has to guess function boundaries from branch
instructions, which works but produces less readable output.

Check these decompilation projects for your game:

| Game | Decompilation Project |
|------|----------------------|
| The Wind Waker | [zeldaret/tww](https://github.com/zeldaret/tww) |
| Twilight Princess | [zeldaret/tp](https://github.com/zeldaret/tp) |
| Melee | [doldecomp/melee](https://github.com/doldecomp/melee) |
| Luigi's Mansion | [Cuyler36/luigis-mansion](https://github.com/Cuyler36/luigis-mansion) |
| Pikmin 2 | [projectPiki](https://github.com/projectPiki/pikmin2) |
| Paper Mario TTYD | [NWPlayer123/PaperMario2](https://github.com/NWPlayer123/PaperMario2) |

Even partial symbol maps help enormously. If your game doesn't have one, you can still
proceed — the recompiler will discover functions automatically.

## Step 1: Extract the Game

Extract the DOL (main executable) and all game files from the ISO:

```bash
# Using Dolphin's file system browser, or:
# Extract everything to a directory
mkdir my_game_files
# Copy main.dol and all other files from the ISO root
```

You need:
- `main.dol` — The main executable (PowerPC code)
- All game data files (textures, models, audio, etc.)
- Any `.rel` files (relocatable modules — loaded dynamically by many games)

## Step 2: Run the Recompiler

```bash
# Recompile the DOL into C source files
./gcrecomp_recompiler main.dol --output recompiled/ --symbols symbols.map
```

This produces a directory of `.cpp` files, one per function:

```
recompiled/
  func_80003100.cpp    // Entry point
  func_80003200.cpp    // OSInit
  func_80004000.cpp    // Main loop
  ...
  func_table.cpp       // Maps addresses to function pointers
```

Each function has the signature:
```c
void func_80003100(gcrecomp::PPCContext* ctx, gcrecomp::Memory* mem);
```

## Step 3: Create Your Game Project

Create a new CMake project that links against gcrecomp:

```cmake
cmake_minimum_required(VERSION 3.20)
project(MyGameRecomp LANGUAGES CXX)

# Add gcrecomp as a subdirectory
add_subdirectory(path/to/gcrecomp)

# Collect all recompiled source files
file(GLOB RECOMPILED_SOURCES "recompiled/*.cpp")

# Build the game
add_executable(my_game
    src/main.cpp           # Your launcher (see template below)
    ${RECOMPILED_SOURCES}  # All recompiled functions
)

target_link_libraries(my_game PRIVATE
    gcrecomp_runtime
    gcrecomp_gx
    gcrecomp_audio
    gcrecomp_input
)
```

## Step 4: Write the Launcher

Your `main.cpp` initializes gcrecomp and calls the game's entry point:

```cpp
#include "gcrecomp/runtime.h"
#include "gcrecomp/gx/gx.h"
#include "gcrecomp/audio/audio.h"
#include "gcrecomp/input/input.h"

using namespace gcrecomp;

// Forward-declare the game's entry point (generated by the recompiler)
extern void func_80003100(PPCContext* ctx, Memory* mem);

int main() {
    // Initialize all subsystems
    runtime_init();
    gx::GXInit();
    gx::GXInitBackend(nullptr, 1280, 720);
    audio::audio_init(32000, 2);
    input::input_init();

    // Set where extracted game files live (for DVD reads)
    set_game_root("./game_files/");

    // Set SDA bases (from the DOL header / symbol map)
    g_ctx.r[2]  = 0x80XXXXXX;  // SDA base (game-specific)
    g_ctx.r[13] = 0x80XXXXXX;  // SDA2 base (game-specific)

    // Call the game's entry point!
    func_80003100(&g_ctx, &g_mem);

    // Cleanup
    runtime_shutdown();
    return 0;
}
```

## Step 5: Test and Iterate

The first run will likely hit unimplemented OS functions. The runtime logs these:

```
[FuncTable] Unresolved indirect call to 0x800ABC10
  LR=0x800DEF00  r3=0x80400000  r4=0x00000100
```

For each unresolved call, you have options:

1. **Check if it's an OS/SDK function** — Look up the address in your symbol map.
   If it's something like `OSCreateThread` or `GXSetTevOrder`, gcrecomp probably
   already handles it. Make sure the symbol map is loaded so the addresses bind.

2. **Add a custom hook** — For game-specific functions you want to replace:
   ```cpp
   void my_custom_handler(PPCContext* ctx, Memory* mem) {
       // Your native implementation
   }
   g_func_table.register_func(0x800ABC10, my_custom_handler);
   ```

3. **Let it run** — Some functions will work correctly once their callees are resolved.

## Step 6: Handle Game-Specific Quirks

Every game has its own quirks. Common things to watch for:

- **DVD file paths**: Games use paths like `/res/Particle/Pscene.arc`. Make sure your
  extracted file structure matches.
- **Memory card access**: gcrecomp stubs `CARD*` functions. If the game needs saves,
  implement the stubs to read/write local files.
- **Audio codecs**: Some games use custom audio formats beyond standard DSP ADPCM.
- **REL modules**: Games that dynamically load code need the REL loader implemented.

## Common Problems and Solutions

**"All functions call through but nothing renders"**
→ The GX draw pipeline needs the vertex format to be fully configured. Check that
  `GXSetVtxDesc` and `GXSetVtxAttrFmt` are being called before `GXBegin`.

**"Game crashes in OSInit"**
→ Low memory isn't set up correctly. Check that `init_low_memory()` was called and
  that the arena bounds make sense for your game's DOL layout.

**"Indirect call to 0x00000000"**
→ A function pointer is null. This often means a vtable wasn't initialized. Check that
  the game's constructors ran before the code that uses the vtable.
